# 数据库架构说明

> 本项目采用**微服务独立数据库架构**，每个服务拥有独立的数据库

## 数据库列表

| 数据库名 | 所属服务 | 端口 | 用户名 | 密码 | 包含的表 |
|---------|---------|------|--------|------|---------|
| `openiot_tenant` | tenant-service | 5432 | openiot | openiot123 | tenant, sys_user, RBAC 相关表 |
| `openiot_device` | device-service | 5432 | openiot | openiot123 | device, device_config, alarm_config |
| `openiot_data` | data-service | 5432 | openiot | openiot123 | device_trajectory, (raw_event 在 MongoDB) |
| `openiot_connect` | connect-service | 5432 | openiot | openiot123 | device_session, connection_log |

## 初始化步骤

### 1. 创建数据库和用户

```bash
# 在 PostgreSQL 中执行
psql -U postgres -f infrastructure/sql/init-databases.sql
```

### 2. 授权 Schema 权限

```sql
-- 在每个数据库中执行
\connect openiot_tenant
GRANT ALL ON SCHEMA public TO openiot;
ALTER DEFAULT PRIVILEGES IN SCHEMA public GRANT ALL ON TABLES TO openiot;
ALTER DEFAULT PRIVILEGES IN SCHEMA public GRANT ALL ON SEQUENCES TO openiot;

\connect openiot_device
GRANT ALL ON SCHEMA public TO openiot;
ALTER DEFAULT PRIVILEGES IN SCHEMA public GRANT ALL ON TABLES TO openiot;
ALTER DEFAULT PRIVILEGES IN SCHEMA public GRANT ALL ON SEQUENCES TO openiot;

\connect openiot_data
GRANT ALL ON SCHEMA public TO openiot;
ALTER DEFAULT PRIVILEGES IN SCHEMA public GRANT ALL ON TABLES TO openiot;
ALTER DEFAULT PRIVILEGES IN SCHEMA public GRANT ALL ON SEQUENCES TO openiot;

\connect openiot_connect
GRANT ALL ON SCHEMA public TO openiot;
ALTER DEFAULT PRIVILEGES IN SCHEMA public GRANT ALL ON TABLES TO openiot;
ALTER DEFAULT PRIVILEGES IN SCHEMA public GRANT ALL ON SEQUENCES TO openiot;
```

### 3. 启动服务（Flyway 自动创建表）

```bash
# 启动各服务，Flyway 会自动执行迁移脚本
cd backend/tenant-service
mvn spring-boot:run

cd backend/device-service
mvn spring-boot:run

cd backend/data-service
mvn spring-boot:run

cd backend/connect-service
mvn spring-boot:run
```

## 跨服务数据关联

### 关联关系

```
tenant (openiot_tenant)
  └─> device.tenant_id (openiot_device)
       └─> device_trajectory.device_id (openiot_data)
       └─> device_session.device_id (openiot_connect)
```

### 实现方式

**❌ 不能使用外键**（跨数据库）

**✅ 应用层实现**：

```java
// device-service 创建设备时，检查租户是否存在
public void createDevice(DeviceDTO dto) {
    // 调用 tenant-service API 检查租户
    Tenant tenant = tenantClient.getTenant(dto.getTenantId());
    if (tenant == null) {
        throw new BusinessException("租户不存在");
    }

    // 创建设备
    deviceMapper.insert(dto);
}
```

## 优势

### ✅ 服务独立

- 每个服务拥有独立数据库
- 符合微服务架构原则
- 可以独立部署、扩展

### ✅ 数据隔离

- 租户数据与其他数据分离
- 安全性高
- 权限控制清晰

### ✅ 故障隔离

- 单个数据库故障不影响其他服务
- 提高系统可用性

### ✅ 独立扩展

- 可以针对不同服务特点优化数据库
- 例如：data-service 数据量大，可以单独扩展

## 注意事项

### ⚠️ 跨服务事务

**问题**：跨数据库事务无法使用本地事务

**解决方案**：
1. **应用层校验**（推荐）：调用其他服务 API 验证数据
2. **分布式事务**（Seata）：需要额外配置和运维成本
3. **最终一致性**（Saga 模式）：适合异步场景

### ⚠️ 数据迁移

从共享数据库迁移到独立数据库：
1. 备份原数据库
2. 创建独立数据库
3. 拆分数据到各数据库
4. 修改应用配置
5. 测试验证

### ⚠️ 联表查询

**问题**：不能跨数据库 JOIN 查询

**解决方案**：
1. **应用层聚合**：分别查询后合并
2. **数据冗余**：允许一定的数据冗余
3. **事件同步**：通过 Kafka 同步数据

## 维护指南

### 新增表

1. 确定表所属服务
2. 在对应服务的 `db/migration/` 创建迁移脚本
3. 重启服务，Flyway 自动执行

### 修改表结构

1. 在对应服务创建新的迁移脚本
2. 版本号递增
3. 重启服务

### 数据备份

```bash
# 备份所有数据库
pg_dumpall -U postgres > backup_all.sql

# 备份单个数据库
pg_dump -U postgres openiot_tenant > backup_tenant.sql
pg_dump -U postgres openiot_device > backup_device.sql
pg_dump -U postgres openiot_data > backup_data.sql
pg_dump -U postgres openiot_connect > backup_connect.sql
```
