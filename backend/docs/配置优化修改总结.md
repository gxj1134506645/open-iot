## 修改内容

### 1. 优化 application.yml（所有服务）

**修改前的问题**:
- application.yml 和 application-dev.yml 中有大量重复配置
- 环境特定配置（数据库、Redis、Kafka）混在公共配置中

**修改后的结构**:
```yaml
application.yml          # 只包含公共配置（所有环境共享）
application-dev.yml      # 开发环境特定配置
application-test.yml     # 测试环境特定配置
application-prod.yml     # 生产环境特定配置
```

**移除的重复配置**:
- ❌ 数据库连接信息（url、username、password）
- ❌ Redis 连接信息（host、port、password、database）
- ❌ Kafka 连接信息（bootstrap-servers）
- ❌ MongoDB 连接信息（uri）

**保留的公共配置**:
- ✅ 服务端口和名称
- ✅ Nacos 配置
- ✅ Gateway 路由配置（仅 gateway-service）
- ✅ Netty 配置（仅 connect-service）
- ✅ 驱动类名（driver-class-name）
- ✅ MyBatis Plus 配置
- ✅ Sa-Token 配置（仅 gateway-service 和 tenant-service）
- ✅ Flyway 配置（仅 tenant-service）
- ✅ Actuator 配置
- ✅ 可观测性配置

### 2. 优化 logback-spring.xml（所有服务）

**修改前的问题**:
- Root 日志级别为 INFO，过滤了 DEBUG 日志
- Spring 框架日志级别过高，启动信息不显示
- 日志输出到控制台的量太少

**修改后的优化**:

#### 开发环境 (dev, local)
```xml
<springProfile name="dev,local">
    <!-- 降低 Spring 框架日志级别 -->
    <logger name="org.springframework.boot" level="DEBUG"/>
    <logger name="org.springframework.web" level="DEBUG"/>
    <logger name="org.springframework.cloud" level="DEBUG"/>
    <logger name="org.springframework.data" level="DEBUG"/>
    <logger name="org.springframework.security" level="DEBUG"/>
    <logger name="org.springframework.kafka" level="DEBUG"/>

    <root level="DEBUG">
        <appender-ref ref="ASYNC_PRETTY"/>
    </root>
</springProfile>
```

**效果**:
- ✅ 显示完整的 Spring Boot 启动日志
- ✅ 显示详细的 Web 请求处理日志
- ✅ 显示数据源初始化日志
- ✅ 显示 Kafka 消费日志
- ✅ 显示 MyBatis Plus SQL 日志

## 配置层级结构

### 日志级别优先级（从高到低）

1. **logback-spring.xml** 中的 `<logger>` 配置
   ```xml
   <logger name="com.openiot" level="DEBUG"/>
   ```

2. **application-{profile}.yml** 中的 `logging.level.*`
   ```yaml
   logging:
     level:
       com.openiot: DEBUG
   ```

3. **application.yml** 中的 `logging.level.*`
   ```yaml
   logging:
     level:
       root: INFO
   ```

### 配置加载顺序

1. application.yml (公共配置)
2. application-{profile}.yml (环境配置，覆盖公共配置)
3. logback-spring.xml (日志配置)

## 常见问题

### Q1: 日志还是很少怎么办？

**可能原因**: 异步 Appender 缓冲区未满

**解决方案**: 等待几秒，或者临时改为同步输出
```xml
<appender-ref ref="PRETTY_CONSOLE"/>  <!-- 同步 -->
```

### Q2: 如何查看完整的启动日志？

**方法 1**: 等待应用完全启动（可能需要 5-10 秒）

**方法 2**: 在 IDEA 中增加控制台缓冲区
- Settings → Editor → General → Console
- 增加 Console history size

**方法 3**: 查看日志文件（如果配置了文件输出）

### Q3: 生产环境日志级别为什么是 INFO？

**原因**:
- 减少日志量，降低 I/O 开销
- 降低存储成本
- 只记录重要信息

**链路追踪**: 通过 traceId 可以定位问题，不需要大量 DEBUG 日志

## 后续建议

### 1. 统一管理日志级别

可以创建一个配置中心（如 Nacos Config）统一管理日志级别，无需重启服务即可动态调整。

### 2. 添加日志文件输出

在生产环境中，建议同时输出到文件：
```xml
<appender name="FILE" class="ch.qos.logback.core.rolling.RollingFileAppender">
    <file>logs/application.log</file>
    <rollingPolicy class="ch.qos.logback.core.rolling.TimeBasedRollingPolicy">
        <fileNamePattern>logs/application.%d{yyyy-MM-dd}.log</fileNamePattern>
        <maxHistory>30</maxHistory>
    </rollingPolicy>
</appender>
```

### 3. 集成日志收集系统

在生产环境中，建议将 JSON 格式的日志发送到 ELK（Elasticsearch + Logstash + Kibana）或 Loki 进行集中管理。
